Respuesta ejercicio 1:

Para pausar el commit actual e ir a borrar el archivo status.js que provoca problemas en producción usaria el "git stash"
estando parado sobre la rama actual (custom-navbar), en caso de tener archivos no trackeados por git podría opcionalmente
 adherirle la opcion "--include-untracked" del comando stash. Esto me permitiría mantener mis cambios en el stash,
 moverme de rama sin problemas, solucionar el issue y volver luego hacia custom-navbar para continuar con el desarrollo
 de esa funcionalidad. Cuando esté nuevamente sobre custom-navbar puedo recuperar los cambios stasheados con "git stash pop".


Respuesta ejercicio 2:
Para pasar a producción el desarrollo que trabajó el equipo de front primero hay que hacer un pull request desde la rama front-react hacia la rama dev,
 una vez aprobada la PR, hecho el merge en dev y las pruebas correspondientes se puede sacar una rama con el prefijo "release" y el numero de versión
 correspondiente (parado en dev: "git checkout -b release/1.0.0"). A partir de esta rama se crea un tag (del ultimo commit, usando 'git tag -a v1.0.0 -m "1.0.0"'),
 que es el que irá a master. Una vez hecho el tag, se pushea hacia el repo (en caso de no haberlo hecho desde el repositorio remoto) 
con "git push origin --tags". Después se hace el PR hacia master de esa versión. Ademas en caso de que dentro de la rama release se hayan hecho cambios, deberia volver
a mergearse a dev para mantener dev actualizada.
